# 大分トリニータ戦評ブログ 生成・管理プロジェクト

## プロジェクト概要

`draft.md` に書かれた試合の感想メモを読み込み、
FOOTBALL LAB 等のデータで感想の主張を検証・補強・反証し、
1500〜2000字のブログ記事（Markdown）を自動生成するツール。

出力ファイル: `articles/YYYYMMDD_相手チーム名.md`
中間ファイル: `research_cache/YYYYMMDD_相手チーム名_research.md`

---

## 起動方法

```
claude
```

Claude Code を起動したら、以下の指示を与える。

```
draft.md を読んで、ブログ記事を生成してください。
```

---

## エージェント構成

以下の3フェーズを順に実行する。各フェーズの完了後、次フェーズに進む前に
中間出力を `research_cache/` に保存すること。

```
[Phase 1] Research Agent  … draft.md の主張を読み取り、データで検証
[Phase 2] Writer Agent    … データ＋感想メモから記事を執筆
[Phase 3] Editor Agent    … ファクトチェック・トーン調整・整形・保存
```

---

## Phase 1: Research Agent

### Step 1-A: 主張の抽出（draft.md の解析）

draft.md は自由形式（なぐり書き・SNS引用・メモ混在）で書かれている。
まず以下のルールで「検証可能な仮説」に変換する。

#### 抽出ルール

- 感情表現 → 背後にある事実仮説に読み替える
  - 例）「プレッシャーが全く効かなくなった」
        → 仮説:「後半、大分のプレス成功率・奪取CBPが低下した」
  - 例）「前半30分まではトリニータのペース」
        → 仮説:「前半序盤は大分のポゼッション・攻撃CBPが優位だった」
  - 例）「赤い選手がわんさかいてトリニータの選手がいなかった」
        → 仮説:「後半、大分の自陣への侵入・守備CBPが悪化した」

- 引用ポスト（X等）も同様に仮説化する
- 「期待・願望」は除外（検証できないため）
  - 例）「後半は立て直してくれよ」→ スキップ

#### Step 1-A の出力形式

```markdown
## 抽出された仮説リスト

1. 仮説: [仮説の内容]
   根拠となった原文: 「[draft.md からの引用]」

2. 仮説: [仮説の内容]
   根拠となった原文: 「[draft.md からの引用]」
```

---

### Step 1-B: データ検索（仮説ごとに検証）

抽出した仮説リストに対して、**支持・反証データの両方**を Web Search で収集する。
都合のいいデータだけ集めてはならない。

#### 検索対象（優先順）

1. **FOOTBALL LAB**（football-lab.jp）
   - チャンスビルディングポイント（CBP）：攻撃・守備・奪取・シュート・パス
   - ポゼッション推移（時間帯別）
   - チームスタイル指標（サイド攻撃・カウンター・ポゼッション種別）
   - 個人CBPランキング
   - マッチレポート URL 形式: `https://www.football-lab.jp/{チームコード}/report?year=YYYY&month=MM&date=DD`

2. **Jリーグ公式**（jleague.jp）
   - スタッツ・順位表

3. **Sofascore / footystats**（補完用）

#### 検索クエリ設計ルール

- 試合単体だけでなく「シーズン推移」「他チーム比較」「直近N試合平均」も探す
- クエリ例:
  - `大分トリニータ football-lab マッチレポート YYYY年MM月DD日`
  - `大分トリニータ CBP 2026 シーズン平均`
  - `J2 守備CBP ランキング 2026`

#### Step 1-B の出力形式

```markdown
## 検証結果

### 仮説1: [仮説の内容]
- 支持データ: [データ・数値・出典URL]
- 反証データ: [データ・数値・出典URL]
- 判定: ✅ 支持 / ❌ 反証 / △ 部分的

### 仮説2: ...

## 取得データ一覧

| 指標 | 今節値 | 今季平均 | リーグ内順位等 |
|------|--------|---------|--------------|
| ...  | ...    | ...     | ...          |
```

#### Step 1-B 完了後の処理

`research_cache/YYYYMMDD_相手チーム名_research.md` に保存してから Phase 2 へ進む。

---

## Phase 2: Writer Agent

### 役割

Research Report と draft.md を統合し、記事本文を執筆する。

### 記事構成（固定）

```
# [タイトル]（感情より分析を示唆するタイトル）

**YYYY年J2第N節 大分トリニータ X-X 相手チーム（日付 スタジアム）**

---

## 試合概要（3〜4行）
スコア・対戦相手・会場・得点者

## 感想の検証（メイン 800〜1000字）
draft.md の感想を起点に、データで掘り下げる。
支持されたなら「なぜそうなったか」を分析。
反証されたなら「見た目と数字が乖離した理由」を考察。

## データで見る今節の大分（300〜400字）
主要スタッツを表形式でまとめ、簡潔なコメントを添える。

## 次節への展望（200字程度）

---

*データ出典: FOOTBALL LAB（football-lab.jp）*
```

### トーンルール

- 一人称は使わず、体言止め・です・ます調を混在させてよい
- 感情的表現は draft.md から引用する場合のみ使用
- データの解釈は断定せず「〜と読める」「〜を示唆している」
- 読者は大分サポーターを想定。専門用語（CBP・xG等）には一言説明を添える

### 文字数

1500〜2000字（データ表・引用・出典行は字数カウント外）

---

## Phase 3: Editor Agent

### 役割

Writer の出力を以下の観点でレビューし、問題があれば修正してから保存する。

### チェックリスト

- [ ] 数値の出典が明記されているか（「FOOTBALL LAB調べ」等）
- [ ] データを捏造・推測で補っていないか（データが見つからない場合は「データなし」と明記）
- [ ] draft.md に書いていない事実を追加していないか
- [ ] 支持データと反証データが両方ある場合、片方だけ使っていないか
- [ ] 見出し構成が上記テンプレートに沿っているか
- [ ] 1500〜2000字の範囲に収まっているか
- [ ] タイトルが内容を適切に表しているか

### 修正後の処理

1. `articles/YYYYMMDD_相手チーム名.md` に保存
2. ターミナルに以下を表示して終了:

```
✅ 記事生成完了
ファイル: articles/YYYYMMDD_相手チーム名.md
文字数: XXXX字
データ引用数: XX件
出典: FOOTBALL LAB マッチレポート YYYY-MM-DD
```

---

## 制約・禁止事項

- データが見つからない場合は「データなし」と明記し、推測で補わない
- FOOTBALL LABのデータは必ずURLを出典として記載する
- draft.md に書いていない試合（別節）のデータを混入させない
- 記事の主語は常に「大分トリニータ」視点
- 他チームへの過度な批判・揶揄は書かない
